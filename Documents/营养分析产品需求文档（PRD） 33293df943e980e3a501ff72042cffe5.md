# 一、概述（为什么做）

## 1.1 产品概述及目标

### 1.1.1 背景介绍

现代人对膳食健康的关注日益增加，健身人群、糖尿病患者等需要精确追踪每日营养摄入。主流营养追踪应用（如 MyFitnessPal）依赖用户手动搜索和录入食材，操作繁琐、误差较大，主要体现在两点：其一，手动录入容易遗忘；其二，份量估算依赖用户主观判断，准确性较低。

现有的图像识别方案大多基于"菜肴级别"识别（如识别出"炒饭""披萨"后查固定营养值），无法反映实际食材种类和用量的差异。

### 1.1.2 产品概述

食材营养识别系统（Ingredient Nutrition Estimator，简称 INE）是一款基于计算机视觉的营养估算工具。用户拍摄烹饪前的**生鲜食材图片**，系统自动识别食材种类、估算重量，并输出各食材及整图的宏量营养素（碳水、蛋白质、脂肪）与总热量，无需手动录入。

系统由 Python 后端（YOLOv8 模型 + FastAPI）和 Web 前端两部分组成，当前版本（V1.0.0）聚焦核心识别与营养计算功能。

### 1.1.3 产品目标

**✏️ 业务目标**

- 近期（课程截止日 2026/04/24）：完成包含食材识别、重量估算、营养计算、Web 可视化展示在内的完整可运行 Demo，满足课程评分要求。
- 远期：扩展至移动端 App，支持更多食材类别（肉类、水果等），面向健身人群和慢性病患者提供日常膳食追踪服务。

**✏️ 用户目标**

- 用户希望用一张照片在 3 秒内得到食材的营养概览，减少手动查询和估算的时间成本。
- 用户希望结果具备基本可信度，能作为日常饮食参考（不要求克级精度）。

### 1.1.4 目标用户

| 用户群体 | 使用场景 | 核心诉求 |
| --- | --- | --- |

# 二、产品描述（做什么）

## 2.1 产品需求描述

系统接收一张包含若干生鲜蔬菜食材的 RGB 图片，输出以下内容：

1. 每种食材的识别类别与置信度；
2. 每种食材实例的像素级分割掩码与边界框；
3. 每种食材的估算重量（克）；
4. 每种食材的宏量营养素（碳水、蛋白质、脂肪）及热量；
5. 全图所有食材的营养总计；
6. 带标注的结果图像（可视化）。

支持的食材类别为 10–15 种常见蔬菜（具体类别待数据集评估后确定，候选包括：甜椒、番茄、胡萝卜、黄瓜、西兰花、菠菜、洋葱、土豆、玉米、南瓜等）。

## 2.2 产品整体流程

### 2.2.1 主流程

`用户上传图片
    │
    ▼
前端调用 POST /predict
    │
    ▼
后端图像预处理（resize → 640×640）
    │
    ▼
YOLOv8m-seg 推理（检测 + 分割）
    │
    ├─► 未检测到食材 → 返回空结果，提示"未识别到支持的食材"
    │
    ▼
各实例：计算面积比 → 估算重量 → 查营养表 → 计算 macros
    │
    ▼
汇总全图营养总计
    │
    ▼
生成标注图像（绘制 mask + 标签）
    │
    ▼
返回 JSON 响应（含 base64 标注图）
    │
    ▼
前端渲染结果（标注图 + 营养表格 + 饼图）`

### 2.2.2 子流程

**子流程 A：重量估算**

`输入：实例分割掩码
    │
    ▼
计算掩码像素面积 Ai
    │
    ▼
计算全图面积 Aimage
    │
    ▼
面积比 ri = Ai / Aimage
    │
    ▼
查询该类别参考基准重量 Wref,i（来自 reference_weights.py）
    │
    ▼
估算重量 wi = ri × Wref,i（单位：克）`

**子流程 B：营养计算**

`输入：wi（估算重量）、类别名
    │
    ▼
查询 nutrition_table.csv（每100g营养值）
    │
    ▼
Ni = (wi / 100) × nper100g,i
    │
    ▼
累加所有实例 → 全图营养总计`

# 三、功能需求（怎么做）

## 3.1 食材识别模块（Level 1）

### 3.1.1 描述

系统对输入图片执行目标检测，输出图中每种食材的类别、置信度和边界框坐标，并自动查询本地营养表返回该食材每100g的营养值。

### 3.1.2 用户故事

- 作为用户，我希望上传一张放有多种蔬菜的图片，系统能准确识别出每种食材的名称。
- 作为用户，我希望看到每种食材对应的基础营养信息（每100g），以便我了解食材的营养密度。

### 3.1.3 前置条件

- 图片格式为 JPG 或 PNG；
- 图片分辨率不低于 300×300 px；
- 图片中食材为平铺放置于平面（切菜板/桌面），非烹饪后菜肴；
- 食材类别在系统支持的预定义列表内。

### 3.1.4 后置条件

- 返回所有检测到的食材实例列表（类别 + 置信度 + bbox）；
- 对每种检测到的类别查询 `nutrition_table.csv`，返回每100g营养值；
- 生成带 Bounding Box 标注的结果图像。

### 3.1.5 界面及交互

（由前端负责，后端提供数据接口，见 API 设计章节）

### 3.1.6 业务流程

1. 接收图片（HTTP POST）；
2. 图像预处理：resize 至 640×640，归一化；
3. 调用 YOLOv8m 推理，得到检测结果（bbox + class + confidence）；
4. 过滤置信度 < 0.5 的结果；
5. 对每个检测结果，按类别名查询 `nutrition_table.csv`；
6. 在原图上绘制 Bounding Box 和类别标签；
7. 将结果图像编码为 base64；
8. 构造 JSON 响应返回。

### 3.1.7 异常/分支流程

- 图中无任何食材被检测到：返回空列表，附带提示信息；
- 检测到的类别不在营养表中：跳过营养查询，标注"营养数据缺失"；
- 同一类别多个实例：分别列出每个实例（不合并）。

### 3.1.8 数据字典

**请求**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| image | file | 上传的图片文件，JPG/PNG |

**响应（`/predict` Level 1 版本）**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| ingredients[].class | string | 食材类别名（英文，如 bell_pepper） |
| ingredients[].confidence | float | 置信度，0–1 |
| ingredients[].bbox | int[4] | [x1, y1, x2, y2]，像素坐标 |
| ingredients[].per100g.calories | float | 每100g热量（kcal） |
| ingredients[].per100g.carbs_g | float | 每100g碳水（g） |
| ingredients[].per100g.protein_g | float | 每100g蛋白质（g） |
| ingredients[].per100g.fat_g | float | 每100g脂肪（g） |
| annotated_image_base64 | string | 带标注的结果图，base64 编码 |

## 3.2 重量估算与营养计算模块（Level 2）

### 3.2.1 描述

在 Level 1 基础上，将检测模型升级为 YOLOv8m-seg，获得每个食材实例的像素级分割掩码。基于掩码面积与全图面积的比值，结合各类别预定义的参考基准重量，估算每个实例的克重，进而计算具体营养摄入量。

### 3.2.2 用户故事

- 作为用户，我希望系统不仅告诉我图中有什么食材，还能告诉我大概有多少克、含多少热量，这样我可以直接用于膳食规划。

### 3.2.3 前置条件

- 满足 3.1.3 所有前置条件；
- 图片中食材为**同一平面、相近距离**拍摄（保证面积比估重有效）；
- 各类别参考基准重量（`reference_weights.py`）已配置完毕。

### 3.2.4 后置条件

- 返回每个食材实例的估算重量（g）；
- 返回每个食材实例的宏量营养素及热量；
- 返回全图所有食材的营养总计；
- 生成带分割掩码标注的结果图像。

### 3.2.5 界面及交互

（由前端负责，后端提供数据接口）

### 3.2.6 业务流程

1. 接收图片，执行预处理；
2. 调用 YOLOv8m-seg 推理，得到 bbox + class + confidence + mask；
3. 对每个实例：
    - 计算掩码像素面积 Ai；
    - 计算 ri = Ai / (图片宽 × 图片高)；
    - 查询 `reference_weights.py` 得到 Wref,i；
    - 估算重量 wi = ri × Wref,i；
    - 查询营养表，计算 Ni = (wi / 100) × nper100g,i；
4. 汇总所有实例营养值得到 totals；
5. 绘制分割掩码标注图；
6. 构造完整 JSON 响应返回。

### 3.2.7 异常/分支流程

- 同一实例掩码面积为0（异常分割）：跳过该实例，记录警告日志；
- 估算重量超过合理范围（如单个食材 > 2000g）：截断至上限，标注"估重异常"；
- 多个同类食材实例：分别独立计算，不合并。

### 3.2.8 数据字典

**响应（`/predict` Level 2 完整版本）**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| ingredients[].class | string | 食材类别名 |
| ingredients[].confidence | float | 置信度 |
| ingredients[].bbox | int[4] | 边界框坐标 |
| ingredients[].mask_area_ratio | float | 面积比 ri |
| ingredients[].estimated_weight_g | float | 估算重量（g） |
| ingredients[].calories | float | 该实例热量（kcal） |
| ingredients[].carbs_g | float | 碳水（g） |
| ingredients[].protein_g | float | 蛋白质（g） |
| ingredients[].fat_g | float | 脂肪（g） |
| totals.calories | float | 全图总热量（kcal） |
| totals.carbs_g | float | 全图总碳水（g） |
| totals.protein_g | float | 全图总蛋白质（g） |
| totals.fat_g | float | 全图总脂肪（g） |
| annotated_image_base64 | string | 带掩码标注的结果图，base64 |

# 四、非功能需求（注意事项）

## 4.1 安全与合规需求

- 用户上传的图片仅用于当次推理，服务端不持久化存储任何用户图片；
- API 不收集用户个人信息，无需处理 GDPR 等合规问题（当前版本）；
- 生产环境部署时应启用 HTTPS。

## 4.2 统计需求

当前版本（课程 Demo）不设埋点统计。后续版本可记录：

- 每日请求量；
- 各类别识别频率；
- 平均推理耗时。

## 4.3 性能需求

| 指标 | 目标值 |
| --- | --- |
| 单次推理响应时间（GPU） | ≤ 3 秒 |
| 单次推理响应时间（CPU） | ≤ 10 秒 |
| 支持图片最大尺寸 | 10 MB |
| 并发支持 | 单实例，不考虑高并发（Demo 场景） |

## 4.4 数据库设计

当前版本不使用数据库，所有数据通过本地文件管理：

- `data/nutrition_table.csv`：食材营养数据（每100g），字段：`class_name, calories, carbs_g, protein_g, fat_g`；
- `nutrition/reference_weights.py`：各类别参考基准重量字典，字段：`{class_name: weight_g}`；
- `configs/classes.yaml`：支持的类别列表。

## 4.5 系统集成

当前版本为独立系统，集成需求仅为：

- 后端 FastAPI 服务需对前端暴露 `POST /predict`、`GET /classes`、`GET /health` 三个接口；
- 接口格式以本文档第三章数据字典为准，前后端对齐后不得单方面修改字段名；
- 标注结果图以 base64 字符串传输，前端负责解码展示。
